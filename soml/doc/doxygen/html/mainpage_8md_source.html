1 OML: The O? Measurement Library {#mainpage}
2 ===============================
3 
4 OML is an instrumentation tool that allows application writers to define
5 customisable measurement points inside new or per-existing applications.
6 Experimenters running the applications can then direct the *measurement
7 streams* (MS) from these *measurement points* (MP) to storage in a
8 remote measurement database.
9 
10 The OML reporting chain consists in several elements:
11 1. the reporting library (C, Ruby or Python),
12 2. the collection server,
13 3. the optional proxy, for disconnected experiments.
14 
15 This documentation mainly focuses on configuring the server, and
16 instrumenting applications. Please refer to file [INSTALL](@ref install) for
17 instructions on building these programs.
18 
19 
20 Running the Server
21 ------------------
22 
23 The [oml2-server(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-server.1.html) receives measurements streams in the OML protocol
24 (either binary marshalling or text mode), and stores them in a database
25 backend. Currently, SQLite3 is the default, and can be replaced
26 PostgreSQL.
27 
28 ### SQLite
29 
30 SQLite [sqlite] is a lightweight SQL file-based database system. Each
31 database is stored as an independent file in the filesystem. The
32 [oml2-server(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-server.1.html) uses this backend by default.
33 
34 Relevant command line arguments are the following:
35 * `-D DIR` or `--data-dir=DIR` specifies to create the database files in `DIR`;
36 * `--user=UID` and `--group=GID` specify as which user (resp. group) the
37  server is run on the system and, consequently, who owns the created
38  database files.
39 
40 Running the server as
41 
42  $ oml2-server -D /path/to/my/databases --user=oml2 --group=oml2
43 
44 will instruct it to drop privileges to user:group `oml2:oml2`, and
45 create and update databases in `/path/to/my/databases`.
46 
47 ### PostgreSQL
48 
49 The database backend can also be PostgreSQL [pgsql]. In this case,
50 additional connection information must be passed to the server so it can
51 properly connect to the server, and create databases.
52 
53 First, a user, say `oml`, with the appropriate permissions has to be
54 created within PostgreSQL. Note that this username has nothing to do
55 with the system user discussed in the previous section.
56 
57  $ createuser -U DBADMIN --pwprompt --no-superuser \
58  --createdb --no-createrole --no-replication oml
59 
60 `DBADMIN` is any pre-existing user allowed to create roles in the
61 database. PostgreSQL's default `postgres` will usually work. The
62 `--no-replication` argument is only valid with PostgreSQL-9.2. In case
63 of error, simply remove it. A password for the newly created database
64 user will be requested. Alternatively, the role can be created through
65 and interactive console.
66 
67  $ psql -U DBADMIN
68  psql (9.3.2)
69  Type "help" for help.
70 
71  DBADMIN=# CREATE USER oml WITH PASSWORD 'oml';
72  CREATE ROLE
73  DBADMIN=# ALTER ROLE oml WITH CREATEDB;
74  ALTER ROLE
75 
76 It is now possible to start the server. Several Postgre-specific
77 options are available (see [INSTALL](@ref install) for details on compiling the PostgreSQL
78 backend in).
79 
80 * `--pg-user=USER` specifies the name of the user just created (default:
81  `oml`);
82 * `--pg-pass=PASS` specifies the password of the user just created
83  (default: empty);
84 * `--pg-host=HOST` specifies the host on which the database server runs
85  (default: `localhost`;
86 * `--pg-port=PORT` specifies the service or port on which the database
87  server expects connections (default: 5432).
88 
89 Of course, it is also necessary to specify that a different database
90 backend is used. This is done with the `-b postgresql` option.
91 Altogether, the server can be run as
92 
93  $ oml2-server --user=oml2 --group=oml2 -b postgresql \
94  --pg-user=oml --pg-pass=correcthorsebatterystaple \
95  --pg-host=db.example.com --pg-port=postgresql
96 
97 ### Server Hook
98 
99 The collection server has a rudimentary support for hook scripts to
100 perform some actions when specific conditions occurs. Currently, only
101 database-close and server-exit are supported. The former can be use for,
102 e.g., backup of measurement databases. Example script
103 server/oml2-server-hook.sh uses the `DBCLOSED` event to push experiment
104 databases to IRODS [irods].
105 
106 ### Integration in Distributions
107 
108 Depending on your distribution, there are various ways to change the
109 [oml2-server(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-server.1.html)'s command line arguments to fit it to your needs.
110 
111 #### Debian
112 
113 In Debian-based system, file `/etc/default/oml2-server` can be modified
114 to adjust the parameters of the server. The server can then be
115 (re)started with:
116 
117  $ sudo service oml2-server restart
118 
119 #### ArchLinux
120 
121 ArchLinux uses systemd(1) as its basic system process. A service unit
122 configuration is provided with the package in AUR [aur]. It is installed
123 in `/usr/lib/systemd/system/oml2-server.service` and uses the SQLite
124 backend in `/var/oml2`. If you want to override these parameters (e.g.,
125 the command line), you need to copy this file to `/etc/systemd/system/`,
126 and modify that copy. Once done, you'll need to refresh systemd's
127 configuration to take the override into account:
128 
129  $ sudo systemctl reenable oml2-server.service
130 
131 You can then (re)start the service with its new parameters.
132 
133  $ sudo systemctl (re)start oml2-server.service

135 Running the Applications
136 ------------------------
137 
138 [liboml2(1)](http://oml.mytestbed.net/doc/oml/2.12/liboml2.1.html) extends the command-line parameters of an instrumented
139 application with options specific to reporting. Most of these options
140 can also be passed as environment variables.
141 
142 OML has the concept of an *experimental domain*, within which several
143 *applications* run on one or more *node*. They generate measurement
144 traffic which is collected and stored together by the [oml2-server(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-server.1.html).
145 Each experimental domain has its own database, with measurements from
146 all the nodes and applications collected there. More details on these
147 concepts are available at [oml-gif].
148 
149 The most important command line arguments (and environment variables) for an
150 instrumented application are the following.
151 
152 * `--oml-id` (or `OML_ID`) is the identifier of the node;
153 * `--oml-domain` (or `OML_DOMAIN`) is the name of the experimental
154  domain;
155 * `--oml-collect` (or `OML_COLLECT`) is a URI pointing to the collection
156  server;
157 * each application also defines its own name (see next section).
158 
159 If `--oml-collect` is not given, a file will be created in the current
160 directory containing text-mode OML protocol [oml-textmode], which can be
161 fed into a server at a later time.
162 
163 Applications can also be configured more finely through the use of a
164 [liboml2.conf(5)](http://oml.mytestbed.net/doc/oml/2.12/liboml2.conf.5.html) file.
165 
166 More details on these topics are available at [oml-conf].

169 Instrumenting an Application
170 ----------------------------
171 
172 Instrumentation of an application (new or pre-existing) is done through
173 [oml2-scaffold(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-scaffold.1.html). Based on an concise description of the desired
174 measurement points, this helper generates the relevant header files and
175 appropriate functions to inject measurement samples into a measurement
176 stream.
177 
178 Essentially, instrumenting an applications therefore consists in writing
179 the application description. A template for this file can be generated
180 for application APPNAME with
181 
182  $ oml2-scaffold --app APPNAME
183 
184 After adjusting this template to contain the desired schemas for the
185 various MPs of the application, a header containing helper functions is
186 then generated with
187 
188  $ oml2-scaffold --oml APPNAME.rb
189 
190 This creates file `APPNAME_oml.h` which can be directly included in the
191 application source to provide the oml_inject_MPNAME(3) helpers.
192 
193 If writing an application from scratch, a template main() function and
194 popt(3)-based option parsing code can also be generated with the
195 `--main` and `--opts` options.
196 
197 It is not recommended to manually change the `APPNAME_oml.h` nor
198 `APPNAME_popt.h` as they are automatically generated. It is best to
199 include rules for the build system to generate them dynamically as part
200 of the build (`BUILT_SOURCES`, in the Automake linguo). As a matter of
201 fact, [oml2-scaffold(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-scaffold.1.html) can also generate a rudimentary Makefile which,
202 based on APPNAME.rb and APPNAME.c, can generate all other necessary
203 files, and build the application. This is done with the `--make`
204 option.
205 
206 See the example, and its documentation in [example/liboml2/README](@ref example), as
207 well as the [oml2-scaffold(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-scaffold.1.html) and oml_inject_MPNAME(3) manpages for more
208 details.
209 
210 Some native implementations of the text mode are also available. Though
211 they are built from this same source tree, it might be easier to install
212 them directly from the language-specific module repositories.
213 * Python: OML4Py [oml4py];
214 * Ruby: OML4R [oml4r];
215 * Java: OML4J [oml4j], maintained by NITLab [nitlab];

217 Using the Proxy Server
218 ----------------------
219 
220 The [oml2-proxy-server(1)](http://oml.mytestbed.net/doc/oml/2.12/oml2-proxy-server.1.html) can be used for situations where a connection
221 to the collection server is temporarily unavailable. It will cache
222 measurement streams until instructed to forward them to the actual
223 collection server.
224 
225 The proxy has two main states: paused (at startup) and running. It
226 expects instructions on its standard input to change states:
227 `OMLPROXY-PAUSE`, `OMLPROXY-RESUME`.
228 
229 Its most important command line arguments allow to specify the
230 collection server's address (`-a ADDR` or `--dstaddress=ADDR`) and port
231 (`-p PORT` or `--dstport=PORT`).
232 
233 Some more details are available at [oml-proxy].

236 Reporting Bugs and Contributing to Development
237 ----------------------------------------------
238 
239 External contributions are most welcome. Be they in the form of bug
240 reports or feature request on the tracker [oml-tracker]. You can also
241 register to the mailing list [oml-ml], where you should direct all your
242 questions, comments or contributions. The archives can be consulted at
243 [oml-ml-archives].
244 
245 The official source tree is stored in a Git repository [oml-git]. It can
246 be cloned without restrictions. To re-integrate patches (fixing bugs or
247 introducing new features), the best way is to make the Git clone
248 containing the changes public, and submit a pull request.
249 
250  $ git request-pull origin/release/2.x http://url/of/your/repo \
251  # Assuming you started working from origin/release/2.x
252 
253 Patches formatted by Git are also a good way to do so.
254 
255  $ git format-patch origin/release/2.x..HEAD \
256  # Assuming you started working from origin/release/2.x
257 
258 In either cases requests and patches should be sent to the mailing list
259 [oml-ml], or a ticket created about them in the issue tracker
260 [oml-tracker].

263 Installing from Distribution Packages
264 -------------------------------------
265 
266 Though you can build OML from source (see [INSTALL](@ref install)), pre-built packages
267 for a few distributions are already available.
268 
269 ### Debian and Ubuntu
270 
271 An archive is maintained on the OpenSuSE Build System, with packages for the
272 latest Debian and Ubuntu distributions. You can add it to your
273 `/etc/apt/sources.list` as
274 
275  deb http://download.opensuse.org/repositories/home:/cdwertmann:/oml/xUbuntu_12.10/ ./
276 
277 then run
278 
279  sudo apt-get update
280  sudo apt-get install oml2-server liboml2
281 
282 to install the required components. You'll need `liboml2-dev` if you
283 want to build instrumented applications.
284 
285 ### ArchLinux
286 
287 ArchLinux packages are available from the Arch User Repository [aur].
288 You can build and install the latest package directly with Yaourt
289 [aur-yaourt].
290 
291  $ sudo yaourt -S oml2
292 
293 The stable package is available at [aur-oml2], while a development
294 version can be obtained from [aur-oml2-git].
295 
296 ### RPM-based (CentOS, Fedora)
297 
298 RPM packages are also built on the OpenSuSE platform. They are available
299 there [opensuse-oml], as well as the instructions to install them on
300 your system.
